DIYAGRAM 3 - Tablo ve Iliski Haritasi

Schema: chat_service

+--------------------------------------------------+
| chat_participants                                |
|--------------------------------------------------|
| PK user_id (UUID)                                |

| email    (unique, not null, idx)                 |
| profile_picture_url (nullable)                   |
| created_at                                       |
+-----------------------------+--------------------+
                              ^
                              | FK creator_id
+-----------------------------+--------------------+
| chats                                            |
|--------------------------------------------------|
| PK id (UUID)                                     |
| creator_id -> chat_participants.user_id          |
| created_at                                       |
+-----------------------------+--------------------+
                              |
                              | M:N via cross ref
                              v
+--------------------------------------------------+
| chat_participants_cross_ref                      |
|--------------------------------------------------|
| chat_id -> chats.id                              |
| user_id -> chat_participants.user_id             |
| unique idx(chat_id,user_id)                      |
| unique idx(user_id,chat_id)                      |
+--------------------------------------------------+

+--------------------------------------------------+
| chat_messages                                    |
|--------------------------------------------------|
| PK id (UUID)                                     |
| content (not null)                               |
| chat_id -> chats.id                              |
| sender_id -> chat_participants.user_id           |
| created_at                                       |
| idx(chat_id, created_at DESC)                    |
+--------------------------------------------------+

Neden bu model guzel?
- M:N katilimci modelini dogru kuruyor.
- Mesaj gecmisi sorgusu icin dogru index var.
- Chat'e giris cikis sorgulari icin join table indexleri iyi.

Teknik borc adayi:
- ChatMessageEntity yaziminda `sender_id` insert akisinin netligi zayif

10) YENI BASLAYAN ICIN ADIM ADIM ORNEK SENARYO
====================================================================
SENARYO A - "Kullanici kendi chat listesini aciyor"

1) Endpoint'e istek gelir (gelecekte eklenecek)
2) requestUserId cekilir
3) ChatRepository.findAllByUserId(requestUserId)
4) ChatEntity listesi doner

SENARYO B - "Kullanici belirli chat detayini aciyor"

1) chatId path'ten gelir
2) requestUserId cekilir
3) findChatById(chatId, requestUserId)
4) donen sonuc:
   - null: kullanici odada degil
   - entity: kullanici odada

SENARYO C - "Mesaj gonderme" (henuz yok)

Beklenen hedef akis:
1) Kullanici odanin participant'i mi kontrol et
2) Mesaj entity olustur
3) sender_id, chat_id, content yaz
4) Son aktivite bilgisini guncelle
5) Realtime/event dagitim yap

12) AKTARIM ICIN HAZIR KONU$MA METNI (KISA)
====================================================================
"Chat modulunde once veri modeli ve DB tarafi cikarilmis.
Uc temel entity var: participant, chat ve message.
Chat ile participant iliskisi many-to-many ile join table'da tutuluyor.
Repository katmaninda en kritik nokta, yetki kontrolunun SQL seviyesinde
EXISTS ile yapilmasi. Bu guvenli bir temel.
Ama su an service/controller/realtime gibi ust katmanlar yok.
Yani bugun elimizde calisan temel altyapi var; bir sonraki adim bu altyapi
uzerine uygulama akislarini insa etmek."
